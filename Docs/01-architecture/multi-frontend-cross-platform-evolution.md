# 多前端与跨平台演进说明

**文档编号**：ARCH-E001  
**版本**：1.0.0  
**状态**：有效  
**最后更新**：2026-04-23  
**维护人**：Am

---

## 1. 文档目标

本文档用于作为 `Docs/` 体系中的上位演进说明，明确以下边界：

1. 当前仓库中的 WinForms / WPF / `.NET Framework` 相关文档主要描述**当前事实**或**当前分支基线**；
2. 这些文档不应被解读为对未来实现路线的长期禁令；
3. 仓库后续允许新增新的 UI 主线、后端形态、宿主方案和运行时框架；
4. 未来版本允许将当前 `.NET Framework` 主线整体迁移并全部替换为 `.NET 10`。

---

## 2. 当前事实与文档边界

截至本文更新时，仓库中的主要现状为：

1. 当前活跃桌面 UI 主线为 `AMControlWinF`；
2. 当前运行框架以 `.NET Framework 4.6.1` 为主；
3. 当前文档中存在大量围绕 WinForms、AntdUI、`MainWindow`、`PageModel`、`RuntimeContext`、`AppBootstrap` 的实现说明；
4. WPF 分支当前主要作为历史实现和对照资料保留。

这些内容的正确理解方式是：

1. 它们描述的是**当前仓库事实**；
2. 它们中的“当前基线”“默认做法”“当前主线”只在对应实现线内部成立；
3. 它们不自动推导为未来所有版本都必须继续沿用的技术路线。

---

## 3. 明确允许的演进方向

### 3.1 新增 UI 主线

仓库后续允许新增新的 UI 主线，包括但不限于：

1. 新的 Windows 桌面主线；
2. Web 前端主线；
3. 跨平台桌面 UI 主线；
4. 面向调试、监控、运维的独立薄前端或远程控制台。

新增 UI 主线时，不要求继续沿用当前 WinForms 的：

1. `MainWindow + UserControl + PageModel` 组织方式；
2. AntdUI 主题系统与控件体系；
3. 当前导航缓存、页面生命周期和布局模板；
4. 当前桌面壳层的交互结构。

### 3.2 新增后端形态

仓库后续允许新增或替换当前后端形态，包括但不限于：

1. 继续使用当前单进程本地后端；
2. 拆分为本地服务 + 前端壳层；
3. 演进为远程 API / WebSocket / gRPC 后端；
4. 在必要时重组运行时、扫描、配置加载、协议接入和设备上报链路。

这意味着当前围绕 `AM.DBService`、`RuntimeWorker`、`Context`、`UsageUploadWorker` 的结构说明，默认只描述当前实现，而不排斥未来重组。

### 3.3 新增宿主方案

仓库后续允许采用新的宿主方案，包括但不限于：

1. 单机桌面宿主；
2. 桌面壳层 + 本地后台服务；
3. 设备侧 Agent + 远程管理端；
4. 面向运维、调试、培训、仿真的独立宿主程序。

因此，当前文档中出现的“主窗体”“当前宿主”“当前壳层”等词，默认仅指当前实现中的宿主，不构成未来唯一宿主形式。

---

## 4. 关于 `.NET Framework` 与 `.NET 10` 的明确声明

关于运行时框架，本文明确声明：

1. 当前文档中涉及 `.NET Framework 4.6.1`、`.NET Framework 4.7.2`、`App.config`、旧式项目文件、`packages.config`、WinForms 设计器等内容，均属于当前实现线事实；
2. 未来版本允许对当前 `.NET Framework` 主线进行部分迁移，也允许进行**整体迁移**；
3. 若业务、部署、性能、跨平台或生态原因需要，未来版本允许将当前 `.NET Framework` 相关实现**全部替换为 `.NET 10`**；
4. 发生此类迁移时，不要求继续维持 `.NET Framework` 与 `.NET 10` 的长期双轨并存；
5. 一旦整体替换完成，现有 `.NET Framework` 文档应转为历史实现说明或归档说明，而新的 `.NET 10` 主线应建立自己的分支级总览、开发基线和进展文档。

这意味着：

1. `AMControlWinF` 当前的目标框架，不代表未来版本的长期承诺；
2. 当前所有围绕 `.NET Framework` 的兼容说明，不代表未来必须继续保留旧框架；
3. 若未来全面迁移到 `.NET 10`，这属于允许且合规的演进方向，而不是“偏离文档约束”。

---

## 5. 不应被误读为长期禁令的内容

以下几类现有文档表述，应统一理解为“当前实现语境下成立”：

1. 某个组件是“当前主入口”“当前主来源”“当前基线”；
2. 某套页面模板、主题系统、导航系统、生命周期是“当前默认做法”；
3. 某条启动链路、运行链路、消息链路、配置链路是“当前稳定实现”；
4. 某些“不建议”“不采用”“不保留”的实现方式。

它们的含义是：

1. 在当前分支继续增量开发时，优先尊重现有结构；
2. 但未来若出现新主线、新框架或大重构，可以建立新的基线文档覆盖这些做法；
3. 不需要为了满足旧文档而强行保留旧壳层、旧框架或旧宿主。

---

## 6. 新主线或框架替换发生时的文档要求

若后续新增 UI 主线、后端形态、宿主方案，或将 `.NET Framework` 整体替换为 `.NET 10`，建议至少同步以下文档：

1. 新增对应的分支级项目总览文档；
2. 新增对应的开发基线或页面模板文档；
3. 新增对应的持续开发进展文档；
4. 更新 `Docs/README.md` 的高价值入口；
5. 更新 `01-architecture/README.md` 的架构索引；
6. 将旧主线文档改为“当前非活跃实现”或“历史实现/归档”口径。

如果发生 `.NET 10` 全量替换，还应额外同步：

1. `02-development` 中的开发环境与兼容性说明；
2. `05-operations` 中的部署与运行环境要求；
3. `08-third-party` 中与运行时、NuGet、打包方式相关的说明；
4. 与框架版本强相关的功能文档和示例请求体。

---

## 7. 当前结论

`Docs/` 体系当前的目标不是把 WinForms、WPF、`.NET Framework`、当前后端结构写成不可变的长期规则，而是：

1. 真实记录当前实现；
2. 允许在未来新增新的 UI 主线、后端形态和宿主方案；
3. 允许未来版本将当前 `.NET Framework` 主线整体替换为 `.NET 10`；
4. 通过新增新的分支级文档，而不是强行修改旧文档，来承载新的实现路线。
