# 文档维护与长期演进规则

**文档编号**：GOV-002  
**版本**：1.0.0  
**状态**：有效  
**最后更新**：2026-04-23  
**维护人**：Am

---

## 1. 文档目标

本文档用于约束 `Docs/` 目录下文档在长期维护中的表述方式、角色边界和同步规则，重点解决以下问题：

1. 同一主题同时存在“实现文档”“规划文档”“进展文档”时，如何判断当前事实；
2. 文档与代码并行演进时，哪些文件必须同步更新；
3. 如何避免把临时协作偏好、阶段性计划写成长期事实；
4. 如何让索引文件在长期维护中保持稳定可用。

---

## 2. 文档角色分层

`Docs/` 目录下文档默认按以下角色理解：

| 类型 | 目录 | 主要用途 | 稳定性要求 |
|------|------|----------|------------|
| 治理文档 | `00-governance` | 规定文档命名、维护规则、长期使用方式 | 最高 |
| 架构文档 | `01-architecture` | 描述长期有效的分层边界、核心约束、生命周期、稳定接口 | 高 |
| 开发文档 | `02-development` | 描述当前活跃分支的开发基线、工程环境、协作方式 | 高 |
| 功能文档 | `03-features` | 描述某项功能的当前实现、接口语义或历史规划 | 中高 |
| 测试/运维/用户手册 | `04-06` | 面向特定使用场景的操作说明 | 中高 |
| 发布记录 | `07-release-notes` | 区分正式发布记录与持续开发进展 | 中 |

---

## 3. 当前事实的来源优先级

当文档之间出现冲突时，按以下顺序判断当前事实：

1. **代码实现**：具体以当前仓库代码为准；
2. **当前活跃实现基线文档 / 当前活跃进展文档**：
   - 现阶段对应 `02-development/winf-project-overview.md`
   - 现阶段对应 `07-release-notes/winf-development-progress.md`
3. **功能实现说明文档**：例如 `03-features/plc-communication.md`、`03-features/device-management-runtime-flow.md`；
4. **架构/设计文档**：用于解释边界和设计意图，但不覆盖已经改变的代码事实；
5. **规划文档 / 历史文档**：仅作为背景资料，不作为当前实现的唯一事实来源。

补充说明：

1. `.github/copilot-instructions.md` 虽不在 `Docs/` 目录中，但承担仓库级协作说明角色；
2. 该文件涉及解决方案现状时，应与当前活跃实现基线文档和当前活跃进展文档保持一致；
3. 若代码已变更但文档未同步，优先修正文档，不以旧文档回推代码结论。
4. 若后续新增前端分支、跨平台宿主或新的主实现线，应补充新的“分支级总览/进展”文档，而不是继续让 WinForms 文档承担全部事实入口。
5. 若需要判断未来是否允许新增 UI 主线、后端形态、宿主方案或整体迁移到 `.NET 10`，应参考 `01-architecture/multi-frontend-cross-platform-evolution.md`。

## 4. 实现文档与规划文档的并存规则

同一主题允许同时保留“实现文档”和“规划文档”，但必须满足以下约束：

1. 文件标题或说明中要能明确区分“当前实现”与“规划/历史”；
2. 若某主题已有实现文档，则规划文档必须显式说明“当前实现以哪份文档为准”；
3. 规划文档继续保留时，应主要承担历史背景、约束来源、取舍记录的作用；
4. 不再适用的规划内容，应移出“当前实现”口吻，避免写成现状；
5. 索引文件必须优先列出实现文档，再列规划文档。

示例：

1. `plc-communication.md` 负责描述当前实现；
2. `plc-communication-planning.md` 若继续保留，应明确标为规划/历史参考；
3. `device-license-and-reporting.md` 可保留接口联调口径；
4. `device-management-runtime-flow.md` 负责描述当前设备端真实实现。

## 5. 长期稳定表述规则

### 5.1 推荐用语

- `已实现`：代码已落地并可在当前仓库中定位到实现；
- `首版`：已落地但仍需继续收口交互、细节或完整性；
- `占位实现`：已有页面或文件，但内容仍是过渡性占位；
- `规划中`：尚未落地，仅有方案；
- `历史规划`：方案文档保留用于背景对照，不代表当前事实；
- `当前优先级`：阶段性开发顺序，可随实现进度更新。

### 5.2 避免用语

- 避免把“将来可能做”的句子写在实现段落中；
- 避免在长期文档中保留会话式表述，如“用户偏好”“本次先这样”“后面再说”；
- 避免把页面占位、演示布局或草拟计划写成“已完成”；
- 避免让一份索引文件同时承担“现状、规划、日志”三种角色而不区分。

## 6. 索引文件的维护规则

每个目录下的 `README.md` 主要承担“入口索引”职责，建议遵循以下规则：

1. 先说明本目录用途，再列文件索引；
2. 当同一主题有实现文档和规划文档时，索引中优先列实现文档；
3. 文件说明中应写明文档角色，例如“当前实现说明”“接口联调说明”“历史规划文档”；
4. 目录 README 不承担详细实现说明，避免把 README 本身写成第二份总览文档；
5. 根目录 `Docs/README.md` 只保留高价值导航与维护说明，不堆叠过细清单。

---

## 7. 文档同步清单

当某项实现已经明显改变当前仓库事实时，至少检查以下同步面：

1. 对应专题文档是否需要更新；
2. 所在目录的 `README.md` 是否仍正确索引；
3. 当前活跃实现基线文档是否需要更新；现阶段对应 `02-development/winf-project-overview.md`；
4. 当前活跃进展文档是否需要更新；现阶段对应 `07-release-notes/winf-development-progress.md`；
5. `Docs/README.md` 是否需要更新高价值入口；
6. `.github/copilot-instructions.md` 是否仍与仓库级事实一致。

若改动属于以下范围，建议强制同步：

1. 导航结构变化；
2. 页面从占位变为已实现；
3. 启动链路变化；
4. 配置表结构变化；
5. 后台 worker / runtime / protocol 行为变化；
6. 开发环境与兼容性边界变化。

## 8. 发布记录与开发进展的区分

`07-release-notes/` 目录下允许存在两类文档：

1. **正式发布记录**：`release-v*.md`，仅用于已发布版本；
2. **持续开发进展**：如 `winf-development-progress.md`，用于活跃分支的持续状态跟踪。

两者区别如下：

| 类型 | 是否等同正式发布 | 推荐用途 |
|------|------------------|----------|
| `release-v*.md` | 是 | 面向版本交付、升级说明、变更归档 |
| `*-development-progress.md` | 否 | 面向当前分支状态、已完成/剩余工作、当前优先级 |

因此：

1. 不要把持续进展文档写成正式 release note；
2. 不要把正式版本记录承担“当前分支工作台账”；
3. 当某功能尚未发布但已在分支中实现时，应优先更新开发进展，而不是伪造 release 文档。

---

## 9. 推荐维护节奏

为降低文档漂移，推荐按以下节奏维护：

1. 单功能实现发生明显变化时，同步专题文档；
2. 一轮页面/架构/后台链路完成后，同步当前活跃实现基线文档与当前活跃进展文档；现阶段对应 `winf-project-overview.md` 与 `winf-development-progress.md`；
3. 增删重要文档入口时，同步目录 README 与根 README；
4. 当长期协作说明发生变化时，同步 `.github/copilot-instructions.md`；
5. 每次文档更新优先做“减法”和“纠偏”，避免无限叠加新层级或重复描述。
